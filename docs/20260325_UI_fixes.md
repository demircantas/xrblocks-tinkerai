# 2026-03-25 UI Fixes

## Purpose
This note captures the current frontend UI state after a full round of XR usability fixes, rollbacks, and pragmatic stopping points.

The goal is to preserve the current working baseline and document what we intentionally deferred so we can move on to the next actions without losing context.

## Current Stable Baseline
The current accepted baseline is:
- guided flow and debug UI both work
- buttons work again
- panels move again
- screenshot preview persists through prompt confirmation
- debug/flow UI switching works from the visible footer button
- hidden UI shells can still steal some hits after switching, but we are explicitly deferring further work on that for now

This baseline was reached by reverting the stricter panel/raycast filtering changes that broke button interaction.

## What Landed Successfully

### 1. Guided flow is working
The default non-debug experience follows the staged flow:
- `generate`
- `segment`
- `compose`

The user can also move backward between stages.

### 2. Screenshot confirmation behavior is fixed
In generate mode:
- capture is scheduled shortly after push-to-talk begins
- the captured image preview remains visible through prompt confirmation
- the preview is dismissed on confirm or cancel

This behavior should be preserved.

### 3. Debug/flow UI toggle is working correctly
The earlier bug where long pinches anywhere could toggle shells was fixed.

Current intended behavior:
- the shell switch is driven by the visible footer button in XR
- desktop toggle still works through the keyboard shortcut path
- the footer button is the correct visible dev-only affordance for now

### 4. Buttons are 1.5x larger
This landed and should stay.

### 5. Current panel behavior is intentionally the older baseline
Panels are currently back to the older whole-panel dragging behavior.
That means:
- panel movement works again
- dragging is still somewhat inconsistent
- missing a button may still move a panel

This is accepted for now because every attempt to over-constrain panel interaction led to worse regressions.

## What Was Tried And Reverted

### 1. Explicit edge/corner drag handles
We tried custom drag handles for panel movement.

Observed result:
- panels either did not move
- or moved unpredictably
- and the implementation consumed too much time

Decision:
- reverted
- do not continue this line of work right now

### 2. Strict UI raycast filtering
We tried making raycast hits land only on buttons/panel shells while disabling other descendants.

Observed result:
- button highlight and press broke
- hidden-shell interactions became harder to reason about
- multiple small fixes did not restore a reliable state fast enough

Decision:
- reverted to the earlier working interaction baseline

### 3. Aggressive panel drag lockdown
We tried disabling panel dragging and selectively restoring it.

Observed result:
- buttons still broke during some variants
- the overall UX became worse than the original baseline

Decision:
- reverted

## Known Remaining Issues

### 1. Hidden panels may still steal hits after shell switch
We attempted a safe subtree interaction disable for hidden shells.

Current status:
- this still appears to happen in practice
- we are not spending more time on it in this pass

Decision:
- document and defer

### 2. Panel dragging is inconsistent
Current behavior still includes:
- some floating-feeling collisions near panels
- inconsistent drag initiation zones
- occasional accidental panel movement when missing buttons

Decision:
- accepted temporarily
- revisit later with a fresh strategy if needed

## Current Technical Position

### Panel interaction
We now believe XR Blocks panel dragging has some built-in behavior that makes it hard to precisely constrain without deeper custom UI work.

Relevant observations:
- draggable panels are convenient, but imprecise
- trying to partially constrain them caused larger regressions than the original problem
- for now, the old baseline is better than the half-constrained variants

### Raycast policy
We are no longer trying to strictly filter every UI descendant in this pass.

Reason:
- the strict policy directly contributed to broken buttons
- working buttons are more important than perfect hit isolation at this stage

## Accepted Tradeoffs
The project is currently accepting the following tradeoffs:
- panel dragging is imperfect but functional
- hidden shell hit stealing may still occur
- panel hits are not yet as strict as we originally wanted

These are acceptable because the more important user-facing paths now work again:
- stage flow
- generation confirmation
- object selection
- compose transforms
- button interaction

## Suggested Next Actions
Priority should now move away from panel-hit perfection and toward more impactful workflow improvements.

Recommended next items:
1. Brush size controls in segment mode.
2. Lighter guided-flow transform gizmo:
   - move
   - single-axis turntable rotation
   - no scale
3. Gesture-based recall for unreachable objects.
4. Any remaining guided-flow clarity/polish items that do not risk button interaction.

## Do Not Reopen Immediately
Avoid immediately reopening these threads unless there is a strong reason:
- edge/corner panel drag handles
- strict buttons-only raycast filtering
- further panel drag micro-fixes

If panel interaction is revisited later, it should be approached as a fresh design pass rather than another incremental patch on the abandoned approach.

## Success Criteria For Current State
The current UI state is considered good enough to move forward if:
- buttons highlight and press reliably
- panels can still be repositioned
- screenshot confirmation works in generate mode
- debug/flow toggle works from the visible button
- guided flow remains usable end-to-end
